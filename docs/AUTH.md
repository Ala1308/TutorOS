# Authentication

TutorOS uses **Supabase Auth** with email magic links. There are no passwords.
This doc explains how the round-trip works, the production setup, and the dev
escape hatches when email is rate-limited or undeliverable.

## TL;DR

- Sign-in is **server-side only** — the browser never sees the access token.
- The `/auth/callback` route accepts **two** flow shapes; both verify the user
  on the server and set the session cookie.
- For local dev when SMTP is broken / rate-limited, run
  `npm run dev:magic-link -- user@example.com` to mint a one-shot URL.
- All production email links must use the `token_hash` callback URL (see
  [§ Production: dashboard email templates](#production-dashboard-email-templates)).
  We never use the legacy implicit hash-fragment flow.

---

## The two supported flows

`app/auth/callback/route.ts` handles both. Anything else falls through to
`/login?error=missing_code`.

### 1. PKCE — `?code=...`

Used when the link was generated by an SSR client that had already stored a
`code_verifier` cookie on `/login`. This is what `signInWithOtp({ email, options: { emailRedirectTo } })`
produces in our server action.

```
Browser → /login (POST email)
       → server action signInWithOtp(email)
       → Supabase emails: ${SITE_URL}/auth/callback?code=<code>
       → /auth/callback exchangeCodeForSession(code) → session cookie
```

### 2. Token-hash — `?token_hash=...&type=...`

Used by `admin.generateLink` (our `dev:magic-link` script) and by the email
templates we ship in `supabase/templates/`. There is no PKCE state cookie —
the token hash itself is the verifier.

```
Server → admin.generateLink({ type, email })
       → email contains: ${SITE_URL}/auth/callback?token_hash=<hash>&type=magiclink
       → /auth/callback verifyOtp({ token_hash, type }) → session cookie
```

We prefer **token-hash** for production email templates because it is
stateless — the same link works no matter which device opens it, no matter
how long the cookie jar has been cleared. PKCE only works on the originating
device.

### What we do **not** support

The legacy "implicit flow" (`#access_token=...` hash fragment). That URL never
reaches the server, so the cookie is never set. If a link redirects to a hash
fragment, the user lands on `/login?error=missing_code` with a clear message.

## Production: dashboard email templates

Local dev (`supabase start`) reads HTML templates from `./supabase/templates/`,
wired up in `supabase/config.toml`. The **linked remote project does not** —
template editing on remote Supabase is dashboard-only.

After provisioning a new Supabase project, paste each template into:

> Supabase Dashboard → **Authentication** → **Emails** → **Email Templates**

Use the bodies committed under `./supabase/templates/`:

| Template       | File                | Subject                             |
| -------------- | ------------------- | ----------------------------------- |
| Magic Link     | `magic-link.html`   | `Sign in to TutorOS`                |
| Reset Password | `recovery.html`     | `Reset your TutorOS password`       |
| Invite User    | `invite.html`       | `You're invited to TutorOS`         |
| Confirm Signup | `confirmation.html` | `Confirm your TutorOS email`        |
| Change Email   | `email-change.html` | `Confirm your new email on TutorOS` |

The critical line in every template is the link target:

```html
<a
  href="{{ .SiteURL }}/auth/callback?token_hash={{ .TokenHash }}&type=<TYPE>"
></a>
```

Where `<TYPE>` is one of: `magiclink`, `recovery`, `invite`, `signup`, `email_change`.

Without this exact pattern, links fall back to the implicit flow and break
silently in App Router.

## Dev escape hatches

### Email is down / rate-limited

```bash
npm run dev:magic-link -- you@example.com
# optionally land on a specific page after sign-in
npm run dev:magic-link -- you@example.com /leads
```

The script:

1. Calls `supabase.auth.admin.generateLink({ type: "magiclink", email })` with
   the service-role key (from `.env.local`).
2. Prints (and clipboard-copies, on macOS) a `…/auth/callback?token_hash=...&type=magiclink`
   URL straight to your browser. **No email is sent**, so it bypasses
   `over_email_send_rate_limit` entirely.
3. The link is one-time use and expires per your project's OTP TTL.

The script will **not** create users. If `auth.users` doesn't have the email
yet, run `npm run seed:owner -- you@example.com "Your Name"` first (it uses
`admin.createUser`, which also doesn't send email).

### `/login` shows an error after clicking a link

`/login` renders a friendly message for known error codes. The full list lives
in `app/(auth)/login/page.tsx`. Common ones:

| `?error=…`        | Meaning                                                                                                                            |
| ----------------- | ---------------------------------------------------------------------------------------------------------------------------------- |
| `missing_code`    | Link came back without `?code` or `?token_hash`. Almost always means a misconfigured email template still using the implicit flow. |
| `otp_expired`     | Link is older than the OTP TTL (default 1 hour).                                                                                   |
| `invalid_link`    | Already used, or token doesn't match. Request a fresh one.                                                                         |
| `callback_failed` | Something blew up in `/auth/callback`. Check server logs.                                                                          |

### Confirm the cookie

After a successful sign-in, DevTools → Application → Cookies → `localhost:3000`
should show **two** cookies set by `@supabase/ssr`:

```
sb-<project-ref>-auth-token         (HttpOnly, Secure on prod)
sb-<project-ref>-auth-token.0       (chunked when JWT is large)
```

If those are missing, the callback never set the session — almost certainly
because the link returned in implicit-flow form. Check the server logs:
`grep "auth callback" .next/standalone/...` (or wherever you stream pino).

## SMTP options when you're ready

In rough order of effort vs reliability:

1. **Built-in Supabase SMTP** — fine for prototypes, capped at 4/hour per project.
2. **Resend** — verify a domain (`resend.com → Domains → Verify`), then in
   Supabase Dashboard → Auth → SMTP Settings, enter `smtp.resend.com:465`,
   user `resend`, password = your Resend API key. Capacity is per your Resend
   plan, not 4/hour.
3. **Gmail SMTP** — requires an App Password and is rate-limited at the gmail
   account level. OK for personal projects, not for prod.

Whichever you pick, the email-template HTML stays the same — only the
delivery transport changes.

## Sign-out

`signOutAction` in `app/(auth)/login/actions.ts` calls
`supabase.auth.signOut()` then `redirect("/login")`. The middleware will
notice the missing cookie on the next protected route hit and bounce
back to `/login`.

## Threat model notes

- We use `signInWithOtp({ shouldCreateUser: false })` so the magic-link form
  does not auto-provision new users. New users come in only via `seed:owner`
  (initial owner) or an explicit invite flow.
- The `signInAction` deliberately returns the same generic "If that account
  exists, a link is on its way" message regardless of whether the email is
  registered, to avoid email enumeration.
- The `over_email_send_rate_limit` error is the only error code we surface
  specifically — it's actionable and not enumerable (any abuser already knows
  emails are rate-limited).
- Service-role key (`SUPABASE_SERVICE_ROLE_KEY`) is only ever read by:
  - `scripts/seed-owner.ts`
  - `scripts/dev-magic-link.ts`

  Both are dev-only scripts, never imported from app code or API routes.
